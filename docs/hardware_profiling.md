# Hardware Profiling and Constrained Runtime Notes

## Scope

This document describes practical profiling outputs generated by the streaming pipeline and how to interpret them on constrained systems.

## Generated Profiling Artifacts

After `run_all`, the pipeline writes:

- `profiles/operator_profile.csv`: per-chunk operator timings and estimated input bandwidth.
- `benchmarks/streaming_chunks.csv`: chunk-level latency, throughput, and memory observations.
- `reports/pipeline_report.json`: aggregate telemetry and operator profile summary.

## Reproducible Profiling Command

Install optional plotting dependencies first if you want PNG benchmark charts:

```bash
pip install -r requirements-plot.txt
```

```bash
cd "nba_data_preprocessing/task"
python run_pipeline.py \
  --input ../data/nba2k-full.csv \
  --output-dir artifacts_profile \
  --chunk-size 128 \
  --batch-size 256 \
  --max-memory-mb 512 \
  --max-compute-units 0.5 \
  --benchmark-runs 3 \
  --random-seed 42
```

## Operator-Level Breakdown

`operator_profile.csv` includes these columns per chunk:

- `preprocess_s`
- `feature_engineering_s`
- `feature_selection_s`
- `encode_scale_s`
- `estimated_input_bandwidth_mb_s`
- `input_bytes`

Use these fields to identify dominant stages under different `chunk_size` and memory constraints.

## Memory Hierarchy and Cache Effects

The project does not capture PMU counters directly, but practical signals include:

- Increased `encode_scale_s` with larger chunks may indicate cache pressure during one-hot encoding.
- Rising end-to-end latency with stable throughput can indicate memory copy overhead.
- Divergence between bandwidth estimate and throughput can indicate storage or serialization bottlenecks.

## Edge and Constrained Scenarios

For low-memory systems:

- enable `--spill-to-disk`
- reduce `--chunk-size`
- keep `--max-memory-mb` realistic for resident process limits

For CPU-constrained systems:

- lower `--max-compute-units`
- use smaller `--batch-size`
- keep `--n-jobs 1` to avoid contention

## Assumptions

- Timing is wall-clock time measured in Python user space.
- Bandwidth is estimated from DataFrame footprint divided by chunk latency.

## Limitations

- No GPU kernels are profiled.
- No direct quantization path is implemented.
- Energy counters are optional and platform-dependent.
